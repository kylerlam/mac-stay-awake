# Mac Stay Awake — Product Requirements Document

## 1. 文档信息

- 产品名称：Mac Stay Awake
- 产品形态：原生 macOS 菜单栏应用
- 文档状态：MVP 开发基线
- 最后更新：2026-07-11

## 2. 产品概述

Mac Stay Awake 是一个轻量的 macOS 菜单栏工具。用户通过单一主按钮，在系统默认的“正常模式”和“保持唤醒模式”之间切换。

应用的核心价值是：用最少的操作，临时防止 Mac 因用户闲置而进入系统休眠，同时让当前状态一目了然。

### 2.1 问题背景

用户会在 MacBook 上运行 Codex 任务、本地服务、下载、编译或其他长时间后台工作。当 Mac 因闲置自动休眠时，这些任务可能暂停、断开连接或无法按预期完成。用户希望不修改长期系统设置，也不使用命令行，即可临时阻止这类闲置休眠。

原始需求特别关注“锁屏后任务继续运行”和“合上盖子后任务不断开”。两者在 macOS 中不是同一个场景：

- **锁屏，但 Mac 仍处于打开状态：** 属于 MVP 的核心支持场景。保持唤醒开启后，锁屏本身不应导致闲置系统休眠，后台任务应可继续运行。
- **普通合上 MacBook 盖子：** 是原始痛点，但不是 MVP 能通过普通 IOKit 闲置休眠 assertion 可靠保证的场景。合盖休眠受 macOS 和硬件策略管理；只有系统原生支持的特定合盖工作条件才可能继续运行。产品不应显示或宣传一个无法履行的“合盖保证”。

### 2.2 需求演进与交互决策

首次构想是一个小窗口，同时放置“正常模式”和“保持唤醒模式”两个按钮，并在下方以绿色文字显示保持唤醒状态。经过需求收敛后，MVP 改为单按钮切换，原因如下：

- 两种模式互斥，单按钮可以直接表达当前可执行的下一个动作。
- 避免已选模式按钮仍看起来可点、两个按钮选中态不清晰等歧义。
- 保留独立状态文字，并在保持唤醒时使用系统语义化绿色，因此不会损失状态可见性。
- 产品入口进一步收敛为菜单栏弹出内容，而非启动时常驻的普通小窗口。

## 3. 目标与非目标

### 3.1 MVP 目标

- 用户可在菜单栏内一次点击开启或关闭保持唤醒。
- 正常模式与保持唤醒模式有清晰、一致的文字和颜色反馈。
- 应用每次启动都从正常模式开始，不自动恢复上次的唤醒状态。
- 通过 IOKit 的电源管理 assertion 防止闲置引起的系统休眠。
- 在屏幕已锁定但 Mac 未进入其他强制休眠场景时，允许 Codex 任务和本地后台服务继续运行。
- 用户可从菜单栏明确退出应用。

### 3.2 非目标

- 不承诺 Mac 普通合盖后仍继续运行；合盖行为仍受 macOS、硬件和外接设备条件约束。
- 不修改系统的永久能源设置。
- MVP 不包含定时关闭、排程、开机自启、快捷键、通知、历史记录或多种唤醒策略。
- MVP 不包含主窗口、设置窗口、账户、云同步或网络服务。
- 不代替用户或系统主动触发的关机、重启或强制休眠。

## 4. 目标用户与核心场景

### 4.1 目标用户

需要在一段时间内让 Mac 保持运行，且希望使用比命令行或修改系统设置更简单方式的用户。

### 4.2 核心场景

- 用户正在进行长时间下载、编译、渲染或数据处理，希望 Mac 不因闲置而休眠。
- 用户锁定屏幕后，希望 Codex 任务或本地后台服务不因闲置系统休眠而中断。
- 用户希望从菜单栏快速确认当前是否已保持唤醒。
- 任务结束后，用户希望一次点击恢复 macOS 的正常能源管理行为。

## 5. 产品形态与信息架构

MVP 是菜单栏专用应用：

- 使用 SwiftUI `MenuBarExtra` 作为唯一主要交互入口。
- 不在 Dock 中显示应用图标，不在启动时打开主窗口。
- 点击菜单栏图标后，显示紧凑的原生弹出内容。
- 弹出内容只包含当前状态、主切换按钮和“退出 Mac Stay Awake”操作。
- 所有可见文案保持简短，单个菜单项不超过 30 个字符。

## 6. 功能需求

### FR-01 启动状态

- 每次进程启动时，应用必须处于正常模式。
- 启动时不创建防休眠 assertion。
- 不持久化上一次的模式。
- 初始菜单栏图标、状态文字和主按钮均表达正常模式。

### FR-02 开启保持唤醒

- 用户在正常模式点击主按钮后，应用请求创建 IOKit 防休眠 assertion。
- 只有在 IOKit 确认创建成功后，状态才切换为保持唤醒。
- 开启成功后，菜单栏图标、状态指示与主按钮采用 macOS 语义化绿色。
- 同一时间最多只允许一个有效 assertion，连续操作不得泄漏或重复创建。

### FR-03 关闭保持唤醒

- 用户在保持唤醒模式点击主按钮后，应用释放当前 IOKit assertion。
- 释放完成后，界面恢复灰色正常模式。
- 恢复正常模式意味着交还给 macOS 原有能源管理策略，而不是立即使系统休眠。

### FR-04 退出与资源清理

- 菜单中提供“退出 Mac Stay Awake”操作。
- 应用退出前必须释放由自身创建的有效 assertion。
- 进程异常终止时，macOS 会随进程结束清理 assertion；产品不声称在进程结束后继续保持唤醒。

### FR-05 失败处理

- 如果创建 assertion 失败，应用必须保持正常模式，不得显示绿色成功状态。
- 弹出内容显示简短、可理解的错误提示，并允许用户再次尝试。
- 错误信息不包含仅对开发者有意义的原始数值或栈信息。
- 如果释放返回错误，应用记录该错误，并不再把旧 assertion ID 视为可复用的有效资源。

## 7. 状态与交互规格

| 状态 | 状态文字 | 主按钮文字 | 颜色 | 点击结果 |
| --- | --- | --- | --- | --- |
| 正常模式 | 正常模式 | 保持唤醒 | 系统灰 / 次要色 | 请求开启保持唤醒 |
| 切换中 | 正在切换… | 正在切换… | 沿用原状态 | 按钮暂时不可用，防止重入 |
| 保持唤醒 | 正在保持唤醒 | 恢复正常模式 | 系统语义化绿色 | 释放 assertion |
| 开启失败 | 无法开启保持唤醒 | 重试 | 正常模式的灰色 | 重新请求开启 |

交互约束：

- 菜单栏图标本身也要反映当前模式，使用户不打开弹出内容也能识别状态。
- 主按钮是唯一的模式切换入口，其位置与尺寸在两种稳定状态中保持一致。
- 颜色不是唯一的状态信号；必须同时提供文字和/或图标差异。
- 不在单次切换成功时弹出系统通知，避免不必要的打扰。

## 8. UI 与可访问性要求

- 使用 Apple 原生、简约的 SwiftUI 控件、字体、间距和语义化色彩。
- 默认适配 macOS 浅色、深色和增强对比度外观，不硬编码白色背景或固定 RGB 绿色。
- 主按钮、菜单栏图标和状态文本提供合理的 VoiceOver 标签与值。
- 可交互元素支持标准键盘焦点和键盘触发。
- 布局在系统字号放大时不应截断核心状态和操作文案。
- 动画只用于帮助理解状态变化，并尊重“减少动态效果”设置。

## 9. 技术要求

### 9.1 技术栈

- 语言：Swift
- UI：SwiftUI
- 系统集成：IOKit Power Management API
- 平台：macOS

### 9.2 唤醒机制

- 使用 `IOPMAssertionCreateWithName` 创建带有可诊断原因文字的 assertion。
- assertion 类型以“防止闲置系统休眠”为目标，实现时应映射到 IOKit 对应的 prevent-idle-system-sleep 类型，不扩展为保证屏幕常亮。
- 屏幕锁定与屏幕关闭不等于系统休眠；应用不应为了让后台任务运行而强制点亮显示器。
- 使用 `IOPMAssertionRelease` 释放 assertion。
- assertion ID 只由一个专用服务持有，UI 不直接调用 IOKit。
- 服务对外暴露最小的开启、关闭和状态能力，并保证幂等清理。

### 9.3 状态所有权

- 应用级单一状态模型是 UI 的唯一事实来源。
- UI 只在底层 assertion 操作成功后进入相应稳定状态。
- 切换期间屏蔽重复操作，避免并发创建或释放。
- 不使用 `@AppStorage` 持久化当前唤醒状态。

### 9.4 项目结构原则

即使 MVP 规模较小，也应按职责分离，避免把应用入口、UI、状态和 IOKit 调用混在一个 Swift 文件中：

- `App/`：应用入口和菜单栏 Scene。
- `Views/`：弹出内容与可复用视图。
- `Models/` 或 `Stores/`：模式状态与切换协调。
- `Services/`：IOKit assertion 的创建与释放。

## 10. 非功能需求

- 性能：应用闲置时不进行轮询，持续 CPU 占用应接近于零。
- 资源：不因长时间保持唤醒而持续增长内存或 assertion 数量。
- 可靠性：快速反复点击不得造成 UI 状态与 assertion 状态不一致。
- 隐私：MVP 不收集、存储或传输个人数据，不请求网络权限。
- 权限：在系统 API 允许的前提下，核心操作不应要求管理员权限或额外隐私授权。
- 日志：只记录启动、模式切换结果和 IOKit 错误等诊断事件，不记录用户内容。

## 11. 验收标准

### AC-01 默认正常模式

**Given** 应用未运行或上次在保持唤醒时退出  
**When** 用户启动应用  
**Then** 应用显示灰色正常模式，且不存在由应用创建的有效 assertion。

### AC-02 开启成功

**Given** 应用处于正常模式  
**When** 用户点击“保持唤醒”且 IOKit 返回成功  
**Then** 只有一个有效 assertion，菜单栏图标、状态与主按钮变为绿色唤醒表达。

### AC-03 关闭成功

**Given** 应用处于保持唤醒模式  
**When** 用户点击“恢复正常模式”  
**Then** 应用释放 assertion，并恢复灰色正常模式。

### AC-04 开启失败

**Given** IOKit 创建 assertion 返回失败  
**When** 用户尝试开启保持唤醒  
**Then** 应用保持灰色正常模式，显示错误提示和重试操作，不显示任何虚假的成功状态。

### AC-05 防止重入

**Given** 模式切换尚未完成  
**When** 用户再次点击主按钮  
**Then** 新操作不会创建额外 assertion，也不会破坏当前切换。

### AC-06 退出清理

**Given** 应用处于保持唤醒模式  
**When** 用户选择“退出 Mac Stay Awake”  
**Then** assertion 被释放，应用进程结束。

### AC-07 系统外观与可访问性

**Given** 用户切换浅色/深色模式、增强对比度或使用 VoiceOver  
**When** 打开菜单栏内容  
**Then** 核心状态和操作仍可读、可辨识、可操作，且不仅依赖颜色传达状态。

### AC-08 防休眠语义

**Given** 保持唤醒已成功开启  
**When** 系统因无用户输入达到闲置休眠条件  
**Then** 应用的 assertion 阻止闲置系统休眠；测试不将普通合盖后继续运行作为通过条件。

### AC-09 锁屏后台运行

**Given** 保持唤醒已成功开启，且 MacBook 保持开盖  
**When** 用户锁定屏幕并等待超过系统原设的闲置休眠时间  
**Then** 系统不因闲置进入休眠，用于验证的本地后台任务在锁屏期间持续运行。

## 12. 测试与验证计划

- 单元测试：状态模型的开启、关闭、错误与防重入转换。
- 单元测试：使用可注入的 assertion 客户端验证创建/释放调用次数及失败结果。
- 集成验证：开启后使用 macOS 电源管理诊断工具确认 assertion 存在，关闭或退出后确认其消失。
- 手动 UI 验证：启动默认状态、两种模式的图标/文字/颜色、快速点击、退出、浅色/深色与 VoiceOver。
- 手动功能验证：在开盖锁屏状态下等待超过系统闲置休眠时间，确认可观测的本地后台任务仍在运行。
- 手动边界验证：确认普通合盖不被宣传或验收为支持场景。

## 13. MVP 交付范围

- 可构建、可运行的原生 macOS 应用。
- 菜单栏弹出 UI 与完整的单按钮状态切换。
- IOKit assertion 生命周期管理。
- 覆盖核心状态与服务行为的自动化测试。
- 简短 README，说明安装/运行、使用方法和合盖限制。

## 14. 后续版本候选项（不属于 MVP）

- 指定时长后自动恢复正常模式。
- 登录时自动启动。
- 全局快捷键。
- 可选的防止显示器闲置休眠模式。
- 本地化和更完整的设置界面。
